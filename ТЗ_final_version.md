# Техническое задание

## Проект: ProjectTracker

### 1. Название проекта

ProjectTracker — система контроля этапов учебных проектов.

### 2. Цель проекта

Разработать веб-приложение для отслеживания выполнения учебных проектов, в котором учитель может создавать проекты и этапы, а ученики — отмечать прогресс выполнения, прикреплять материалы и обмениваться сообщениями с учителем.

### 3. Описание проекта

В системе предусмотрены две роли: `учитель` и `ученик`.

Учитель создает учебные проекты, добавляет этапы с описанием и дедлайнами, отслеживает прогресс учеников по каждому этапу и может выгружать итоговую таблицу проекта в формате `CSV`. Ученик видит свои проекты, изменяет статус выполнения этапов, прикрепляет ссылку на GitHub-репозиторий, загружает файл и оставляет комментарий. Дополнительно в системе реализуется простая внутренняя переписка между учеником и учителем.

Приложение должно поддерживать несколько проектов одновременно.

### 4. Основной функционал

#### Функционал учителя

- регистрация и авторизация;
- создание проекта;
- создание и редактирование этапов проекта;
- просмотр списка учеников проекта;
- просмотр общей таблицы прогресса;
- просмотр карточки конкретного ученика;
- отправка и чтение сообщений;
- выгрузка итоговой таблицы проекта в формате `CSV`.

#### Функционал ученика

- регистрация и авторизация;
- просмотр списка своих проектов;
- просмотр этапов проекта и дедлайнов;
- изменение статуса этапа;
- добавление GitHub-ссылки;
- прикрепление файла к этапу;
- добавление комментария;
- отправка и чтение сообщений.

### 5. Используемые технологии

- `Python`
- `Flask`
- `Flask-SQLAlchemy`
- `Jinja2`
- `Bootstrap 5`
- `Flask-Login`
- `SQLite`

### 6. Хранение данных

В проекте будут использоваться две базы данных `SQLite`.

#### Основная база `main.db`

Предназначена для хранения основных данных приложения:

- пользователи;
- проекты;
- участники проектов;
- этапы проектов;
- прогресс по этапам.

#### Служебная база `service.db`

Предназначена для хранения служебных данных:

- сообщения между пользователями;
- журналы действий пользователей;
- журналы обращений к API.

### 7. Веб-интерфейс

В приложении будут реализованы:

- страницы регистрации и авторизации;
- личный кабинет учителя;
- личный кабинет ученика;
- страницы проектов и этапов;
- форма изменения статуса этапа;
- возможность прикрепления файла к этапу;
- форма отправки сообщений;
- выгрузка итоговой таблицы проекта в `CSV`.

### 8. REST API

В проекте будет реализован собственный `REST API` с ответами в формате `JSON`.

Планируемые запросы:

- `GET /api/projects` — список проектов;
- `GET /api/projects/<id>/students` — список учеников проекта;
- `GET /api/projects/<id>/stages` — этапы проекта;
- `GET /api/projects/<id>/stats` — статистика по проекту;
- `GET /api/students/<id>/progress` — прогресс ученика;
- `POST /api/messages` — отправка сообщения;
- `POST /api/progress` — создание или обновление данных о прогрессе.

### 9. Ожидаемый результат

Итогом должен стать работающий веб-проект, в котором:

- реализованы роли учителя и ученика;
- поддерживается работа с несколькими проектами;
- данные хранятся в двух базах `SQLite`;
- пользователь может работать через веб-интерфейс;
- учитель может выгружать итоговую таблицу проекта в `CSV`;
- приложение предоставляет собственный `REST API`.

### 10. Минимально рабочая версия проекта

В минимально рабочую версию проекта планируется включить:

- файл `requirements.txt`;

- регистрацию и авторизацию пользователей;
- две роли: учитель и ученик;
- поддержку нескольких проектов;
- создание проектов и этапов;
- изменение статуса этапов;
- прикрепление GitHub-ссылки;
- прикрепление файла к этапу;
- добавление комментария к этапу;
- простую переписку между учеником и учителем;
- выгрузку итоговой таблицы проекта в `CSV`;
- собственный `REST API` с `GET`- и `POST`-запросами.
